Layer 3: Decision (D) — what you’re actually building
Layer 1 gives truth (Context Bundle).
Layer 2 gives evaluations (Judgement Report).
Layer 3 commits to a course of action.
This is the “commit point” layer. If Layer 3 is sloppy, your system becomes:
overconfident
inconsistent
impossible to debug
unsafe for automation
Decision Layer’s job is to output a plan + an execution mode + a trace + a response package.
No real execution happens here either. Execution happens in the Execution layer (tools / agents). Decision only instructs.

Non-negotiables
Fail-closed: if J says missing info or high risk → no auto-exec
Traceable: plan must cite judgement + policies + sources
Composable: planning, mode selection, trace, packaging are separate modules
Predictable: same inputs should produce materially same plan

D0. Decision Router (small but needed)
Purpose
Route to the correct decision template engine based on:
intent_type
judgement constraints
available tools
Output
decision_route
D1. Intent Clarification
Purpose
Convert messy goals into a canonical intent type your system supports.
What it does (MVP)
Choose one intent type
Extract key slots (who, what, when, channel)
Resolve conflicts using judgement signals (missing info, ambiguity)
Subsections
1. Intent Classifier
map to canonical set
2. Slot Extractor
entities + required fields for that intent
3. Intent Confidence Scorer
if low confidence → ask clarifying questions
4. Intent Finalizer
lock intent_type and intent_slots
Output
intent_type.json
intent_slots.json
D2. Plan Generation
Purpose
Generate a concrete action plan with steps, tool calls, and fallbacks.
What it does (MVP)
Create step sequence
Select tools (based on tool_context availability)
Draft tool payloads (but don’t execute)
Layer 3 (Decision Layer)
2Include fallback plan per tool call
Subsections
1. Plan Template Selector
choose a plan template by intent_type + policies
2. Step Builder
build ordered steps with dependencies
3. Tool Call Draft Generator
produce payload drafts for tools
4. Fallback Compiler
what to do when tool fails / rate limited / missing permissions
5. Constraint Enforcer
apply constraints from judgement (policy + risk + multifactor)
Output
action_plan.json
{ steps[], tool_calls[], fallbacks[] }
D3. Approval and Execution Mode Selection
Purpose
Decide “how far we go”:
auto execute
needs approval
propose only
ask clarifying questions
escalate
What it does (MVP)
Hard-gate auto execution using judgement:
policy_verdict
risk class
missing info
tool permissions
Produce an execution mode + rationale
Subsections
1. Mode Gate Engine
deterministic if/else rules
2. Approval Chain Resolver
who needs to approve (role/person)
3. User Interaction Strategy
ask questions vs propose draft vs request approval
Output
execution_mode.json

D4. Explanation and Authority Trace
Purpose
This is your “authority” product layer.
Not fluffy explanation. A trace that can survive scrutiny.
What it does (MVP)
Why this plan
Which policies applied
Which factors dominated (from J5)
What context influenced it (sources)
What was rejected and why (“what not to do”)
Subsections
1. Reason Graph Builder
connect: context → judgement → decision
2. Policy Trace Builder
list matched policies + effects
3. Alternative Rejection Logger
show 1–2 rejected options and why
4. Audit Trail Packager
store trace in decision logs
Output
decision_trace.json
{ why[], policies[], factors[], sources[], rejected_options[] }

save instructions (what to store in memory/logs)
UI blocks (for dashboard rendering)

1. Brain Response Builder
concise message, structured UI blocks
2. Tool Instruction Compiler

How to build this with agentic builders (without writing messy
code)
Build order
1. Define schemas
2. Implement D3 Mode Gate Engine first (safety)
3. Implement D1 Intent Clarification (stability)
4. Implement D2 Plan Templates + Step Builder (core value)
5. Implement D4 Trace (moat)
6. Implement D5 Packaging (integration)
7. Assemble + metrics
Layer 3 (Decision Layer)
6Prompt pattern for Cursor/Codex
For each module instruct:
“pure function only”
“input and output strictly matches schema”
“must include reason strings + source_refs”
“unit tests for 3 scenarios per intent type”
“no tool execution in Decision layer”
Concrete example: follow-up to VIP investor
Inputs
Context: gmail thread, last reply date, user style, VIP tag
Judgement: policy needs approval, risk high, priority high, factors ranked
Decision output
intent_type: follow_up
execution_mode: needs_approval
plan:
1. draft follow-up email using template
2. show preview + reasons
3. request approval from founder
4. once approved, send at recommended window
trace: VIP policy + fundraising mode + last interaction + user tone preference
brain_response: UI blocks: “Draft”, “Why this”, “Approval button”
This is exactly where you start looking like a “brain” instead of “automation.”
The blunt warning
If D2 is “LLM writes steps”, you’ll ship chaos.
Your plan generation must be template + constraints + slot filling, not free-form reasoning.
Reasoning is allowed in trace and small choices, but structure must be enforced.
That’s how you stay reliable and enterprise-ready.
