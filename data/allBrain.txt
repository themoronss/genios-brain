Naive design
Each atom does:

The solution: 5 controls

2) Use a “Two-pass atomization”
First pass generates a small number of macro-atoms, not micro-atoms.
Second pass expands only the macro-atom that is chosen to execute.
Pass 1 (cheap)

judge + pick the best one
Pass 2 (expensive, only for chosen one)
expand into detailed micro steps
draft tool payloads

3) Parallelize only cheap tasks, serialize expensive ones
Don’t parallelize LLM-heavy steps blindly.
Do in parallel

Cache layers

11➡️ CONCEPT CLARIFICATION
First: What the hell are “atoms”?
Forget the word.
An atom = a candidate action path.
That’s it.
When a user says:
Follow up to Investor X tomorrow at 8am
The system does NOT immediately decide one thing.
It first generates possible action paths like:
1. Draft follow-up email and schedule send
2. Ask user clarification (tone / attachment?)
3. Check if investor replied recently before sending
4. Escalate because VIP
Those are macro atoms.
They are not steps.
They are competing high-level strategies.
Why do we need macro atoms?
Because if you go directly into detailed planning:
You waste LLM calls.
You expand 10 paths deeply.
Only 1 will be executed.
So instead:
1. Generate 3–6 macro action strategies
2. Judge them cheaply
3. Pick the best
4. Only then expand into micro steps
That’s what I meant by:
Shared context + selective expansion
So yes — Orchestrator does atomization.
Layers don’t create atoms.
