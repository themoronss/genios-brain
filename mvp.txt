04  The Exact V1 Build — Component by Component


Component 1: Gmail OAuth + Ingestion
Time to build: 1 week

OAuth 2.0 Gmail connection — read-only scope: gmail.readonly
Backfill job: read last 6 months of emails in batches of 100 threads
Show progress bar to user during build

Extract per email:
Sender name + email address
Recipients (to, cc)
Date and time
Subject line
Body text — plain text only, strip HTML
Thread ID to group conversations

NOTE
No sentiment yet. No complex NLP. Just clean structured data. Get this right before adding complexity.


Component 2: Entity Extraction
Time to build: 1 week

Extract per email:
Name — from email header
Email address
Company — from email domain
Role — from email signature if present

Basic classification:
Internal? — same domain as org
Known VC? — check against list of 500 known VC domains
Otherwise — external contact

LLM extraction prompt (GPT-4o mini or Claude Haiku):

"Given this email, extract:
1. One sentence summary of what was discussed
2. Sentiment: positive / neutral / negative
3. Intent: follow_up / request / commitment /
          introduction / negotiation / other
4. Any commitments made (things promised)
5. Key topics discussed
Return as JSON only."

Entity resolution — simple version only:
Same email address = same person (exact match)
Do not build fuzzy matching yet
Unknown = create new node
Duplicates handled manually in V1

DECISION
Do NOT use Neo4j for V1. Too complex to set up and tune quickly. Use PostgreSQL with graph-like queries. Migrate to Neo4j at V2 when you have the data volume to justify it.

Relationship stage logic — simple rules:
Last interaction < 7 days + positive sentiment = ACTIVE
Last interaction 7–30 days = WARM
Last interaction 30–60 days = DORMANT
Last interaction > 60 days = COLD
Sentiment avg < -0.3 = AT_RISK — overrides all above

Recalculate nightly via simple cron job.

Component 4: The Context API
Time to build: 3 days

CORE
One endpoint only. This is the entire product in API form. The context_for_agent field in the response is the magic — a ready-to-use paragraph any agent can prepend to their prompt.


Endpoint:
POST /api/v1/context
{
  "org_id": "uuid",
  "entity_name": "Sarah Chen",
  "situation": "investor follow-up"
}

Returns:
{
  "entity": {
    "name": "Sarah Chen",
    "company": "Sequoia Capital",
    "relationship_stage": "WARM",
    "last_interaction": "12 days ago",
    "sentiment_trend": "positive",
    "communication_style": "concise, data-forward",
    "topics_of_interest": ["retention", "GTM", "India"],
    "open_commitments": ["Send retention data by March 15"],
    "what_works": "Short emails with specific metrics",
    "what_to_avoid": "Long narrative emails",
    "recommended_action": "Send brief update with numbers"
  },
  "context_for_agent": "Sarah Chen is a Partner at Sequoia.
    Last contact 12 days ago — she responded positively.
    She asked for retention data in the Feb 18 call.
    She prefers short, data-forward emails.
    Open commitment: send retention data by March 15.
    Safe to share revenue range, not exact ARR.",
  "confidence": 0.82
}

Zero integration complexity. Massive context improvement.

Component 6: One Agent Integration
Time to build: 3 days

Pick ONE integration only. The simplest possible. Recommended: direct Claude or GPT integration via prompt injection.

The flow:

User: "Draft a follow-up to Sarah Chen about our Series A"

GeniOS step 1: Detect "Sarah Chen" in request
GeniOS step 2: Call /api/v1/context for Sarah Chen
GeniOS step 3: Prepend context to agent prompt

"ORGANISATIONAL CONTEXT:
 Sarah Chen, Partner at Sequoia Capital.
 Relationship: WARM. Last contact 12 days ago.
 She asked for retention data. Committed to send by March 15.
 She prefers short data-forward emails.
 Safe: revenue range. Not safe: exact ARR.

 USER REQUEST:
 Draft a follow-up to Sarah Chen about our Series A"

GeniOS step 4: Agent drafts with full context
GeniOS step 5: Result shown to user

No SDK. No orchestrator. No complex integration. Just context injection via prompt. That is the entire V1 agent integration.

09  The One Thing That Will Make Or Break V1



CONTEXT BUNDLE ACCURACY

If you return context that is wrong — wrong relationship stage, wrong summary of last conversation, wrong tone recommendation — the customer loses trust immediately. And they never come back.

You have one shot at the "how did it know that?" moment. It has to be right.


The Right Way To Build Accuracy
Before you write a single line of code for the extraction pipeline:

1
Manually read 20 email threads from 20 different people across different orgs
2
Write a context bundle by hand for each person — not with code, by hand
3
Note what signals you used to make each judgment
4
Build the code to do what you just did manually
5
Validate: does the code produce what you would have written by hand?


PRINCIPLE
Build the judgment first. Then automate the judgment. Never the other way around. Spend 80% of engineering time making context accurate. Not on features. Not on UI. On accuracy.
