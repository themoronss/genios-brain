## The Exact V1 Build — Component by Component

### Component 1: Gmail OAuth + Ingestion

```
WHAT TO BUILD:
──────────────

Simple OAuth 2.0 Gmail connection
Read-only scope only:
gmail.readonly

Backfill job:
→ Read last 6 months of emails
→ Process in batches of 100 threads
→ Show progress bar to user

What to extract per email:
→ Sender name + email
→ Recipients
→ Date + time
→ Subject line
→ Body text (plain text only, strip HTML)
→ Thread ID (to group conversations)

That is it for ingestion.
No sentiment yet.
No complex NLP.
Just clean structured data.

TECH STACK:
───────────
Google Gmail API
Simple Python backend (FastAPI)
PostgreSQL for raw email storage
Celery for background processing
Redis for job queue

TIME TO BUILD: 1 week
```

### Component 2: Entity Extraction (Simplified)

```
WHAT TO BUILD:
──────────────

For each email extract:

PEOPLE:
→ Name (from email header)
→ Email address
→ Company (from email domain)
→ Role (from email signature if present)

BASIC CLASSIFICATION:
→ Is this internal? (same domain as org)
→ Is this a known VC? (check against list of 500 known VC domains)
→ Otherwise: external contact

INTERACTION SUMMARY:
→ Call GPT-4o mini with this prompt:

"Given this email, extract:
1. One sentence summary of what was discussed
2. Sentiment: positive / neutral / negative
3. Intent: follow_up / request / commitment /
           introduction / negotiation / other
4. Any commitments made (things promised)
5. Key topics discussed

Email: {email_body}

Return as JSON only."

ENTITY RESOLUTION (simple version):
→ Same email address = same person (exact match only)
→ Do not build fuzzy matching yet
→ Unknown = create new node
→ Duplicates handled manually in v1

TIME TO BUILD: 1 week
```

### Component 3: Relationship Graph (Simplified)

```
WHAT TO BUILD:
──────────────

NOT Neo4j for V1.
Too complex to set up and tune quickly.

Use PostgreSQL with graph-like queries.
You can migrate to Neo4j at V2.

TABLES:

entities (people and companies)
──────────────────────────────
id                UUID
org_id            UUID
name              String
email             String
company_name      String
company_domain    String
entity_type       Enum: INVESTOR/CUSTOMER/VENDOR/TEAM/UNKNOWN
relationship_stage Enum: COLD/WARM/ACTIVE/DORMANT
first_interaction Timestamp
last_interaction  Timestamp
interaction_count Integer
sentiment_avg     Float
summary           Text (LLM generated relationship summary)
topics            Text[] (recurring topics)
communication_style String
open_commitments  Integer

interactions
────────────
id                UUID
org_id            UUID
from_entity_id    UUID
to_entity_id      UUID
timestamp         Timestamp
medium            String (email)
direction         Enum: INBOUND/OUTBOUND
sentiment         Float
intent            String
content_summary   Text
topics            Text[]
commitments_made  Text[]
thread_id         String

RELATIONSHIP STAGE LOGIC (simple rules):
→ Last interaction < 7 days + positive sentiment = ACTIVE
→ Last interaction 7-30 days = WARM
→ Last interaction 30-60 days = DORMANT
→ Last interaction > 60 days = COLD
→ Sentiment avg < -0.3 = AT_RISK (override above)

Recalculate nightly. Simple cron job.

TIME TO BUILD: 1 week
```

### Component 4: The Context API

```
WHAT TO BUILD:
──────────────

ONE endpoint only:

POST /api/v1/context
{
  "org_id": "uuid",
  "entity_name": "Sarah Chen",
  "situation": "investor follow-up"
}

Returns:
{
  "entity": {
    "name": "Sarah Chen",
    "company": "Sequoia Capital",
    "role": "Partner",
    "relationship_stage": "WARM",
    "last_interaction": "12 days ago",
    "interaction_count": 14,
    "sentiment_trend": "positive",
    "communication_style": "concise, data-forward",
    "topics_of_interest": ["retention", "GTM", "India market"],
    "open_commitments": ["Send retention data by March 15"],
    "what_works": "Short emails with specific metrics",
    "what_to_avoid": "Long narrative emails",
    "recommended_action": "Send brief update with retention numbers"
  },
  "recent_interactions": [
    {
      "date": "March 1",
      "summary": "Sent Q4 traction update, she replied positively",
      "sentiment": "positive"
    },
    {
      "date": "February 18",
      "summary": "30 min call about GTM strategy",
      "outcome": "positive, requested retention data"
    }
  ],
  "context_for_agent": "Sarah Chen is a Partner at Sequoia
    Capital. Your last interaction was 12 days ago when you
    sent a traction update and she responded positively.
    She specifically asked for retention data in your Feb 18
    call. She prefers short, data-forward emails. You have
    an open commitment to send retention data by March 15.
    Do not mention exact ARR — revenue range is safe.",
  "confidence": 0.82
}

The context_for_agent field is the magic.
It is a ready-to-use paragraph that any
agent can prepend to their prompt.

Zero integration complexity.
Massive context improvement.

TIME TO BUILD: 3 days
```

### Component 6: One Agent Integration

```
WHAT TO BUILD:
──────────────

Pick ONE agent integration only.
The simplest possible one.

RECOMMENDATION: Direct Claude/GPT integration
via simple prompt injection.

Customer pastes their API key.
Customer types a task.
GeniOS fetches context.
Injects into prompt.
Agent responds with full org awareness.

Simple flow:

User: "Draft a follow-up to Sarah Chen about our Series A"

GeniOS:
1. Detects "Sarah Chen" in request
2. Calls /api/v1/context for Sarah Chen
3. Prepends context to agent prompt:

   "ORGANISATIONAL CONTEXT:
   Sarah Chen, Partner at Sequoia Capital.
   Relationship: WARM. Last contact 12 days ago.
   She asked for retention data. You committed to
   send by March 15. She prefers short data-forward
   emails. Safe to share revenue range, not exact ARR.

   USER REQUEST:
   Draft a follow-up to Sarah Chen about our Series A"

4. Agent drafts with full context
5. Result shown to user

That is the complete V1 agent integration.
No SDK. No orchestrator. No complex integration.
Just context injection via prompt.

TIME TO BUILD: 3 days
```

## The Tech Stack For V1

```
BACKEND:
────────
Language:     Python (FastAPI)
Database:     PostgreSQL (single DB, simple)
Cache:        Redis
Queue:        Celery + Redis
LLM calls:   Anthropic Claude API (haiku for extraction,
              sonnet for context bundle generation)
Gmail:        Google API Python client

FRONTEND:
─────────
Framework:    Next.js
Graph visual: react-force-graph (simple, good enough)
Styling:      Tailwind CSS
Auth:         NextAuth.js

INFRASTRUCTURE:
───────────────
Hosting:      Railway or Render (simple, fast to deploy)
Database:     Supabase (PostgreSQL, managed)
Redis:        Upstash (managed Redis)

DO NOT USE YET:
───────────────
✗ Neo4j (overkill for V1 data volume)
✗ Pinecone (no semantic search needed in V1)
✗ Kubernetes (too complex)
✗ Microservices (monolith for V1)
✗ Kafka (no event streaming needed yet)

V1 is a monolith.
Fast to build. Easy to debug.
Microservices at V2 when you know what to separate.
```

## The One Thing That Will Make Or Break V1

```
CONTEXT BUNDLE ACCURACY.

If you return context that is wrong —
wrong relationship stage,
wrong summary of last conversation,
wrong tone recommendation —

The customer loses trust immediately.
And they never come back.

You have one shot at the "how did it know that?" moment.
It has to be right.

So before you build anything else:
Spend two weeks manually reading email threads
and writing context bundles by hand.

Not with code.
By hand.

Do this for 20 different people across different orgs.

Then build the code to do what you did manually.

That is the only way to know
what accurate actually looks like
before you automate it.

Build the judgment first.
Then automate the judgment.
Never the other way around.
```
